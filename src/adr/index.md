# ADR — архитектурные решения

## ADR-001. PostgreSQL как основная БД

**Статус:** принято.  
**Решение:** использовать PostgreSQL для хранения пользователей, вакансий, откликов и интервью.  
**Причина:** предметная область содержит много связей, транзакций и отчётов. PostgreSQL хорошо подходит для реляционной модели и аналитических запросов.

## ADR-002. Выделение Recommendation Service

**Статус:** принято.  
**Решение:** вынести рекомендации в отдельный сервис.  
**Причина:** алгоритмы матчинга могут развиваться независимо от основного Backend API. Это позволяет отдельно тестировать качество рекомендаций и масштабировать расчёт.

## ADR-003. OAuth2/OIDC через SSO университета

**Статус:** принято.  
**Решение:** не хранить пароли студентов, а использовать университетский SSO.  
**Причина:** это снижает риски безопасности и упрощает вход для студентов.

## ADR-004. Модерация вакансий до публикации

**Статус:** принято.  
**Решение:** вакансия не попадает в каталог без проверки модератором или автоматической антиспам-проверки.  
**Причина:** система работает с университетской аудиторией, поэтому качество вакансий важнее скорости публикации.

## ADR-005. Outbox для уведомлений

**Статус:** принято.  
**Решение:** события уведомлений сохраняются и обрабатываются асинхронно.  
**Причина:** сбои e-mail/мессенджера не должны ломать основной сценарий отклика или интервью.
